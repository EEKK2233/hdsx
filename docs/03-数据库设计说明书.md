# 数据库设计说明书

## 2026-07 表结构增补

- `lesson_resources.is_saved`：区分教师主动保存与最多 30 条临时生成历史。
- `questions.material_type`：标记课堂例题、练习题、思考题或拓展材料。
- `qa_messages.original_content/correction_note/corrected_by/corrected_at`：保留 AI 原回答及教师修正证据。
- `notifications`：按用户存储作业截止、答疑修正等站内通知，以 `notification_key` 保证幂等。
- 家长学生关系继续由 `parent_student_links` 隔离，课程操作由 `course_managers` 校验。
- `documents.source_url`：保存确认入库网页的最终来源地址，本地上传文件保持为空。
- `web_import_drafts`：保存网页正文预览、来源、内容哈希、确认状态和 24 小时有效期；迁移版本为 `0008_web_imports`。

## 1. 概述

业务数据库采用官方 MySQL 8.4 Docker 镜像，当前实际服务版本 8.4.10，字符集 `utf8mb4`；数据持久化在 `D:\Docker\mysql`。向量库采用现有 Milvus。MySQL 保存权威业务数据、文档块正文和检索元数据，Milvus 仅保存向量及 `chunk_id`。文件本体保存于 `storage/`。

## 2. 配置与迁移

数据库名默认为 `ai_education`，应用账号为 `ai_education_app`。管理员先执行 `database/init_mysql.sql` 并修改密码，再配置 `.env`。结构通过 `backend/migrations` 管理：

```powershell
conda activate llm_learn
cd backend
alembic upgrade head
```

生产环境禁止应用使用 root。密码应通过环境变量注入。连接启用 `pool_pre_ping`、1800 秒 recycle、默认池 10、溢出 20。

## 5. 关键字段与类型

- 所有主键使用 `BIGINT AUTO_INCREMENT`。
- 外部标识 `public_id` 使用 UUID 字符串，避免暴露连续主键。
- 成绩使用 `DECIMAL(7,2)`，模型置信度使用 FLOAT 且代码限制在 0～1。
- 文档正文、题干和评语使用 TEXT；结构化内容使用 JSON。
- 网页草稿正文使用 MySQL MEDIUMTEXT，应用层限制为 5 万字符；来源 URL 使用 VARCHAR(2000)。
- `lesson_resources.content_json` 继续使用 JSON 列兼容历史版本，新生成内容以 `{title, text, format, resource_type}` 保存，其中 `text` 为可直接阅读和下载的 Markdown 正文。
- 状态使用统一枚举：draft、processing、ready、published、pending_review、approved、failed、archived。
- 时间统一由服务端产生并以 UTC 语义保存。

## 6. 索引策略

- 用户名、课程 owner、文档 course/category、提交 student/assignment、掌握度 student/knowledge_point 建普通索引。
- 文档块对 `content` 建 MySQL FULLTEXT，并保留 LIKE 兜底。
- 使用 `content_hash` 防止块级重复；文档使用 `(course_id, dedup_key)` 唯一约束阻止并发上传相同内容。历史文档的 `dedup_key` 可为空，新上传文件均写入 SHA-256。
- 网页确认时以提取后的正文 SHA-256 同时写入 `file_hash` 和 `dedup_key`，防止相同正文通过不同 URL 重复入库。
- 分页查询统一使用稳定排序；大表后期采用游标分页。

## 7. 事务与一致性

- 文档上传先保存文件和 MySQL 状态，再处理块；异常回滚数据库并保留可诊断文件。
- 网页抓取只写 `web_import_drafts`；确认事务成功后关联 `confirmed_document_id`。拒绝、过期或索引失败的草稿不得成为 ready 文档。
- Milvus 不参与 MySQL 事务。正式向量入库采用状态机和幂等补偿：MySQL processing → 写向量 → 回写 milvus_id → ready。
- 批改结果和提交总分在同一事务写入；教师复核不得覆盖 AI 建议分。
- 报告指标和自然语言版本一起保存，防止内容与后续实时数据不一致。

## 8. Milvus 设计

Collection 默认 `edu_chunks_dev`，向量维度 768（已从当前 embeddinggemma 环境确认）。字段包含主键、chunk_id、course_id、document_id、category、content_hash、embedding 和 created_at。查询必须带 course_id 权限过滤，返回 chunk_id 后回查 MySQL。

## 9. 备份与安全

定期执行 MySQL 逻辑备份和 storage 文件备份，并保持同一恢复点。备份不提交 Git。恢复测试应验证 MySQL 文档块与 Milvus chunk_id 一致性。审计日志和成绩数据设置更长保留期。

## 10. 0009 重复提交与删除语义

`submissions` 新增非空 `attempt_no`，唯一键调整为 `(assignment_id, student_id, attempt_no)`，并增加最新尝试复合索引。课程和作业采用 `archived` 归档删除以保留成绩审计链；知识库文档按负责人确认执行实体删除，先清理向量和 `document_chunks`，再删除文件记录与受控 storage 文件。

## 11. `p` 协作模块数据约束

`p/` 模块原则上不直接持有 SQLAlchemy Session，不得调用 `create_all`、执行任意 DDL 或维护 SQLite/第二套迁移链。需要新增表或字段时必须进入主项目 Alembic 迁移、设计评审和备份流程。抓取记录删除只删除 `web_import_drafts`；已确认生成的 Document 不连带删除。本轮数据库版本仍为 `0009_submission_attempts`。
