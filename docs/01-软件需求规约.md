# 软件需求规约

## 2026-07 功能增补

- 页面切换后保留课程选择、表单和课堂答疑上下文；答疑会话持久化并最多展示最近 30 条。
- 备课结果支持保存、下载、历史查看和主动删除；未保存记录按先进先出保留 30 条。
- 注册支持学生、教师和家长；家长注册必须绑定已有学生，并按学生隔离查看已发布报告。
- 教师可查看全部课程信息，但课程审批、备课、作业、答疑和学情操作仅限其负责课程。
- 作业可从知识库文件按章节生成课堂例题、练习题、思考题和拓展材料。
- 学生与教师均可查看薄弱知识画像和推荐材料；系统汇总提问与错题高频重点。
- 教师可审核并修正 AI 答疑，系统通知学生；学生接收作业临近截止提醒。
- 作业材料生成应兼容模型代码块、顶层数组和常见字段别名；失败时进行一次受控 JSON 修复。
- 多选题支持任意选项组合，判断题固定提供正确/错误；题目在产生学生提交前允许再次编辑。
- 空标准答案由 AI 依据题目与课程知识库补充；学生提交后立即获得建议分、反馈和标准答案。
- 报告数组字段即使收到字符串也不得按字符渲染，应规范化为完整语句或条目。

## 1. 引言

### 1.1 目的

本文定义“AI 教育智能备课与个性化学习辅导智能体”的功能、接口、数据、安全和质量需求，是开发、测试和验收的共同依据。

### 1.2 项目范围

系统服务于教师、学生、家长和管理员，围绕课程知识库形成“资料入库—备课—答疑—作业—批改—学情—报告”的教学闭环。首版为 Vue Web 端和 FastAPI 后端，业务数据存入 MySQL，向量数据存入 Milvus，AI 能力由本地 Ollama 和重排序模型提供。

### 1.3 术语

- RAG：检索增强生成。
- 文档块：长文档经递归字符/token 分割得到的最小检索单元。
- 辅助评分：AI 给出建议分和依据，教师保留最终确认权。
- 掌握度：根据学生答题证据计算的 0～1 知识点指标。

## 2. 总体描述

### 2.1 用户角色

| 角色 | 权限边界 |
| --- | --- |
| 管理员 | 初始化用户、系统管理、查看运行状态 |
| 教师 | 管理本人课程、资料、备课、作业、批改和报告 |
| 学生 | 参与课程、提交本人作业、提问和查看本人学情 |
| 家长 | 仅查看已授权学生且已发布的报告 |

### 2.2 运行环境

- Windows 开发机；Python 使用 Conda `llm_learn`，版本 3.11。
- MySQL 8.x，默认端口 3306。
- Milvus Docker 服务，默认端口 19530。
- Ollama，默认地址 `http://localhost:11434`。
- 模型：`qwen2.5:latest`、`embeddinggemma:latest`、`bge-reranker-v2-m3`。
- 后续部署目标为 Linux；嵌入式/TFLite 属于扩展验证。

### 2.3 约束

- 不得自动降级到 SQLite。
- 不得重复下载现有 Milvus 镜像或修改其已有数据目录。
- 模型不能执行自由 SQL；写操作必须通过后端业务服务。
- AI 评分与报告在发布前必须允许教师审核。

## 3. 功能需求

### FR-01 身份与权限

系统应支持初始化首个管理员、学生自主注册、登录、当前用户查询、资料修改、密码修改、管理员创建用户和基于角色的接口鉴权。自主注册只能创建学生账号；修改密码必须验证当前密码。首个用户创建后初始化接口必须关闭。密码不得明文存储。

### FR-02 课程与知识结构

教师应能创建课程、章节和知识点。一个课程可设置多位负责人，任一负责人可维护课程、导入成员和审批申请。学生可发现课程并提交加入理由，只有审批通过或由负责人导入后才成为课程成员。教师只能操作自己负责的课程；管理员可管理全部课程。课程应支持学科、学段、描述和状态。

### FR-02A 课程学生管理

课程负责人应能上传 CSV 批量导入学生，表头包含 `username`、`display_name`，可选 `email`、`password`；已有学生账号直接加入课程，新账号按名单创建。系统应展示逐行导入结果。学生可以申请加入课程，负责人可以同意或拒绝并填写审批意见。

### FR-03 文档知识库

系统应支持一次选择最多 20 个 TXT、Markdown、PDF、DOCX 文件批量上传。上传时验证文件大小、文件名和内容 SHA-256；同一课程中内容相同的文件即使改名也必须阻止入库，并由 MySQL 唯一约束抵抗并发重复。知识库页面应展示文件名、分类、状态、分块数和上传时间。文档使用递归分割，默认约 650 token、重叠 100 token。每块必须包含 `source`、`filename`、`chunk_index`、`category` 和 `course_id`。

系统通过 MySQL 关键词检索和 Milvus 向量检索执行多路召回，以 RRF 合并去重，再用 BGE 重排序并返回来源。当前版本已完成完整检索链路并通过真实服务验证。

### FR-04 智能备课

教师输入课程、章节、授课对象、课时、资源类型和要求后，系统应检索课程资料并生成结构化 JSON。内容至少包含目标、重点、难点、教学步骤、例题、练习、总结和作业，并保存引用。

### FR-05 作业与批改

教师应能创建作业、添加客观题或主观题并发布，并按课程查看作业列表、详情、提交数量和学生提交队列。学生应看到所有对其发布的作业、课程、标题、总分、题目数、截止时间和个人提交状态，并能进入单个作业填写。学生接口不得返回标准答案或 rubric。客观题采用确定性规则判分；主观题依据标准答案和 rubric 调用模型辅助评分。每项结果包含分数、置信度、评语和评分依据，默认处于待复核状态。

学生只能读取已正式加入课程的知识库、作业和答疑能力；仅发现课程或提交待审批申请不授予数据读取权限。课堂答疑必须以课程名称选择器展示学生已加入或教师负责的课程，不要求用户输入内部课程编号。

### FR-05A 角色化界面

系统导航、总览快捷入口、页面和操作按钮必须根据 admin、teacher、student、parent 角色显示。教师可创建课程、上传资料、创建与批改作业；学生只能查看参与课程、检索资料、填写作业和查看本人学情；家长只查看已授权且已发布的报告。后端必须独立执行同样的权限校验，不能只依赖前端隐藏按钮。

### FR-06 教师复核

教师可调整最终得分和评语。最终得分不得超出题目满分。系统保留 AI 建议与最终结果，成绩发布前完成复核。

### FR-07 课程答疑

学生创建课程问答会话后可提问。系统只依据课程知识库回答并返回引用。无证据时应拒绝编造并标记需要教师处理。

### FR-08 学情分析

系统按学生和知识点保存掌握度、等级、证据数量与算法版本。默认等级为薄弱（<0.60）、待巩固（0.60～0.79）、已掌握（>=0.80）。学生只能查看本人数据。

### FR-09 学习报告

教师选择学生、课程和周期生成周报或月报。指标必须先由数据库计算，LLM 只负责自然语言解释。生成结果为待审核状态，未发布报告家长不可见。

### FR-10 健康与错误

系统应提供存活与就绪检查。MySQL、Ollama 等依赖不可用时，就绪接口返回 503 和依赖错误。业务错误统一返回 `code`、`message`、`details` 和 `request_id`。

## 4. 非功能需求

### NFR-01 安全

密码使用 PBKDF2-HMAC-SHA256 与随机盐；JWT 使用 HMAC 签名并设置到期时间。文件路径必须去除目录部分。日志不得记录密码、令牌和完整学生隐私。

### NFR-02 性能

普通 CRUD 在本地网络目标响应时间小于 1 秒；单次普通检索目标小于 3 秒；模型生成允许 120 秒超时。长任务应逐步迁移到异步队列。

### NFR-03 可靠性

数据库事务失败必须回滚。模型输出必须经过 JSON 和范围校验。外部服务故障不得产生伪造成功结果。

### NFR-04 可维护性

依赖方向为 API → Service → Repository/ORM。配置来自环境变量。数据库变更使用 Alembic。所有模块具有自动化测试或明确验收用例。

### NFR-05 可追溯性

AI 回答保存 `trace_id`、引用、置信度和模型配置；重要业务写操作应记录审计日志。

## 5. 验收标准

使用固定演示课程完成以下流程：初始化账号、教师创建课程、上传教材并检索、生成教案、创建和发布作业、学生提交、AI 批改、教师复核、查询学情、生成待审核报告。自动化测试通过，接口文档可访问，依赖异常有明确提示。
